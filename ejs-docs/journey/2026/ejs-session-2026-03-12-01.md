# Decisions Made
- Decision: Use `EXCEL_MCP_WORKBOOK` and `EXCEL_MCP_SERVER` env vars as the interface between the test script and the CLI, rather than CLI args.
  - Reason: The CLI's positional arg parser interprets the first non-`-` argument as the command; passing paths as args before the command string would require parser changes.
  - Impact: Test script is simpler; env var approach also documents real usage pattern for operators.

- Decision: Fix CLI parameter names (`cell` â†’ `cell_address`, etc.) rather than adjusting test or server.
  - Reason: The CLI had bugs â€” it was sending wrong parameter names to the MCP server. The server's tool signatures are the source of truth (documented in descriptions).
  - Impact: Write operations now work correctly for CLI users beyond just the test script.

- Decision: Change write helper return type to `Task<bool>` and propagate exit code.
  - Reason: CLI tools should exit non-zero on failure. Original void return meant errors were silent to callers (shell scripts, CI).
  - Impact: Proper exit codes enable scripted usage and test automation.

# Key Learnings
- The MCP SDK `ModelContextProtocol` v1.1.0 only supports tools, not resources. Any CLI or client code using `resources/list` will fail at runtime.
- CLI parameter names sent via `JsonObject` to `CallToolAsync` must exactly match the C# parameter names in the `[McpServerTool]`-annotated method (underscored snake_case).
- `dotnet build` on two projects back-to-back can hit a transient MSBuild file-lock on shared Contracts. A 1-second gap resolves it.
- `Task<bool>` return from write helpers is sufficient â€” no need for full `Task<int>` refactor of all methods.

# If Repeating This Work
Do this: Always check CLI `JsonObject` keys against the actual server tool method parameter names before assuming the CLI works.  
Avoid this: Running both builds without a gap in automated scripts.  
Watch out for: MCP SDK version changes that may add or remove protocol capabilities (resources, prompts).

# Future Agent Guidance
Prefer: Verifying MCP tool parameter names by reading `ExcelTools.cs` before writing CLI call code.  
Avoid: Assuming `IsError` on `CallToolResult` causes non-zero exit â€” the CLI must explicitly call `return 1` to propagate errors.
