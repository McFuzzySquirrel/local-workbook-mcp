# Key Learnings
**Technical:**
- A grep sweep (`grep -rn "net9\.0\|localhost:1234"`) is a reliable and fast way to find stale version/URL refs after a major migration — run it as a post-migration checklist step.
- `run-chatweb.sh` health-checks a specific LLM URL; any docs or scripts that reference the LLM should align with whatever URL `run-chatweb.sh` probes, or the check becomes misleading.

**Tooling:**
- `python3 scripts/adr-db.py` (not `python`) on this Ubuntu system.
- Multi-replace tool is highly efficient for batched cross-file fixes — one round-trip for 8 edits.

# If Repeating This Work
Do this:
- Run a grep sweep for hardcoded TFM versions and LLM URLs at the start of any migration session to build the full hit list before touching files.
- Verify with grep after each fix batch, not just at the end.

Avoid this:
- Assuming all files are consistent after a migration — `appsettings` files often have Windows dev paths that carry the old TFM.

Watch out for:
- `run-chatweb.sh` LLM health-check URL must stay in sync with the configured default LLM provider. If the provider changes again, update the health-check URL, all `localhost:*` references, and README Step 3 together.

# Future Agent Guidance
Prefer:
- Delegating broad codebase audits to the `Explore` sub-agent before deciding fix scope — avoids guessing and missing files.
- Using multi-replace for batched cross-file fixes of the same pattern.
- Verifying with grep after each fix batch.

Avoid:
- Assuming `install.sh`/`install.ps1` need changes when updating TFM refs — they use high-level `dotnet publish` commands that resolve the TFM from the project file.
