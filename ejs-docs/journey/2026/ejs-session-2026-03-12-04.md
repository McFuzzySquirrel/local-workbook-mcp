# Decisions Made
- Decision: Clear `ServerPath` to `""` in `appsettings.Development.json` (rely on env var from `run-chatweb.sh`).
  - Reason: Non-empty config shadows env var in `EnsureExcelMcpConfiguration`; `run-chatweb.sh` already sets the correct absolute path.
  - Impact: Developers running via `run-chatweb.sh` get the correct server path automatically.
- Decision: Inject `IWebHostEnvironment` into `McpClientHost` for path resolution.
  - Reason: Makes relative-path resolution explicit and CWD-independent for any paths that do come through config.
  - Impact: Future config paths can be relative to `ContentRootPath` without CWD surprises.
- Decision: Use `OperatingSystem.IsWindows()` conditional for the hardcoded fallback exe name.
  - Reason: Linux/macOS binaries have no `.exe` extension; Windows does.
  - Impact: Cross-platform fallback without separate scripts.
- Decision: Wire `DetectRunningModel()` result back into `config` in SkAgent `Program.cs`.
  - Reason: Detection was already correct but only used for display; the actual kernel was built with the stale default.
  - Impact: SkAgent now uses the detected model for all LLM calls, eliminating the 404 model-not-found error.

# Key Learnings
- `Path.GetFullPath(path)` resolves against `Directory.GetCurrentDirectory()`, which `dotnet run` sets to the project directory — not the repo root. Always pass a base path.
- A non-empty config value silently wins over environment variables in `IConfiguration`. Invalid-but-non-empty settings are a common source of "why is my env var ignored?" bugs.
- `Path.GetFullPath` in .NET does **not** expand `~` — that is shell-only.
- In SkAgent, having `DetectRunningModel()` return the right value but not use it was a classic "detected but not applied" bug — easy to miss because the header showed the right model name.
- When file tools are unavailable, sub-agents without tools cannot implement fixes. Verify tool availability before delegating implementation work.

# If Repeating This Work
Do this: Before trusting a config value, confirm it is non-empty AND valid. Prefer absolute paths or relative paths anchored to a known base.
Avoid this: Using `~` in JSON config or `Path.GetFullPath` without a base in a web app context.
Watch out for: Non-empty invalid config values masking correct environment variable overrides.

# Future Agent Guidance
- In this repo, `dotnet run --project` sets CWD to the **project** directory. Any `Path.GetFullPath` without a base will resolve relative to `src/<ProjectName>/`, not the repo root.
- The pattern for cross-platform executables: `OperatingSystem.IsWindows() ? "name.exe" : "name"`.
- When adding auto-detection that updates a value, always check it's wired into what actually *uses* that value (not just what *displays* it).
