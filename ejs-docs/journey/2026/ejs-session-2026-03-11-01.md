- MCP is now universally supported by Claude Desktop, GitHub Copilot, Cursor — but the server is hand-rolled and won't connect to them

1. Migrate to official MCP SDK (`ModelContextProtocol` NuGet)
2. Add write operations (cell/range write-back with file backup)
3. Ollama as default provider with auto-detection
4. MCP client integration configs (Claude Desktop, GitHub Copilot, Cursor)
5. Streaming responses in Blazor UI

# Decisions Made
- Decision: Migrate to official `ModelContextProtocol` NuGet SDK
  - Reason: Hand-rolled JSON-RPC won't connect to Claude Desktop, GitHub Copilot, Cursor, or VS Code Agent Mode
  - Impact: Unlocks all major MCP clients; replaced ~400 LOC with ~150 LOC
- Decision: Include write-back operations in MVP
  - Reason: Users expect to make changes; read-only is a significant UX limitation
  - Impact: New ExcelWriteService + 3 new MCP tools; file backup for safety
- Decision: Ollama as default LLM provider (LM Studio retained as fallback)
  - Reason: Dominant local runtime in 2026; Docker-friendly; free
  - Impact: appsettings.json defaults changed; auto-detect logic updated
- Decision: Each MCP tool accepts `workbook_path` parameter (dual-mode design)
  - Reason: External clients (Claude Desktop, Copilot) can't pass workbook path at startup; standalone mode uses --workbook/env var
  - Impact: Tools construct ExcelWorkbookService per-call; stateless and flexible
- Decision: Blazor UI stays as standalone experience
  - Reason: MCP client configs are additive; not replacing the web app
  - Impact: Phase 4 (client configs) is purely additive, no Blazor changes
- Decision: `ExcelTools` class is `sealed` (not `static`)
  - Reason: `WithTools<T>()` SDK generic constraint disallows static types
  - Impact: Methods are logically stateless (new ExcelWorkbookService per call); minimal overhead
- Decision: MCP server logging redirected to stderr
  - Reason: MCP stdio protocol uses stdout exclusively; logging must not contaminate the protocol stream
  - Impact: `LogToStandardErrorThreshold = LogLevel.Trace` + `MinimumLevel = Warning` in host config
- Decision: No `ProviderType` enum — cascade auto-detect instead
  - Reason: Simpler and zero-config; most users either have Ollama or LM Studio running, not both
  - Alternatives: `LlmProviderOptions` with enum + explicit selection UI
  - Impact: `DetectProviderAsync` tries `:11434/v1/models`, then `:1234/v1/models`; overrides both model + base URL on success
- Decision: Bump all TFMs from `net9.0` to `net10.0`
  - Reason: Ubuntu 24.04 apt only has .NET 8 and .NET 10 runtimes; `net9.0` test host fails at launch
  - Alternatives: Install .NET 9 runtime from Microsoft feed (extra configuration, non-standard)
  - Impact: All 8 .csproj files updated; build and 68 tests pass
- Decision: Upgrade `Microsoft.SemanticKernel` to 1.73.0 (from 1.26.0/1.28.0)
  - Reason: Critical CVE GHSA-2ww3-72rp-wpp4 discovered via NU1904 during build
  - Alternatives: Continue with vulnerable version (unacceptable per security policy)
  - Impact: No breaking changes; SK 1.73.0 is API-compatible for all usages in this codebase
- Decision: Streaming via in-place mutation of placeholder `ConversationTurn`
  - Reason: No duplicate history entries; Blazor component sees live updates directly; avoids `AddAssistantTurn` adding a second entry after finalization
  - Alternatives: (a) Separate streaming state variable in component — adds state complexity; (b) Replace placeholder at end — causes flash
  - Impact: `StreamQueryAsync` creates and adds the turn, mutates Content per-chunk, finalizes; caller just iterates and calls `InvokeAsync(StateHasChanged)`

# Key Learnings
- `.NET 9 SDK` is not in Ubuntu 24.04 (noble) apt repos — only .NET 8 and .NET 10. .NET 10 SDK builds net9.0 TFM targets fine. But test runners need the matching runtime, so TFMs should match an installed runtime.
- `ModelContextProtocol` v1.1.0 SDK uses `[McpServerToolType]` on the class and `[McpServerTool(Name = "...")]` on methods. Tool class must be non-static (generic constraint in `WithTools<T>()`).
- Dual-mode workbook resolution (env var fallback + per-call param) lets the same server binary work with both `--workbook` CLI flag and external MCP clients that pass the path per tool call.
- `WithStdioServerTransport()` handles all transport framing — no need for custom Content-Length header parsing.
- `NU1904` vulnerability warnings during build are actionable security signals — check and upgrade immediately when detected.
- `GetStreamingChatMessageContentsAsync` with `ToolCallBehavior.AutoInvokeKernelFunctions` works correctly — SK handles tool call round-trips internally and continues streaming the final response.
- Blazor Server streaming pattern: mutate shared state from the streaming loop + `await InvokeAsync(StateHasChanged)` to re-render. `InvokeAsync` ensures we're on the component's dispatcher thread, which is correct after `await`.

# If Repeating This Work
Do this:
- Check dotnet SDK availability before starting (Ubuntu noble has 8 and 10, not 9)
- Use `dotnet package search` to confirm latest NuGet versions before hardcoding
- Make tool classes `sealed` (not `static`) when using SDK `WithTools<T>()`
- Run `dotnet build` after package upgrades to surface NU1904 vulnerability warnings — fix immediately
- For Blazor streaming: mutate state in-place + `InvokeAsync(StateHasChanged)` — do NOT try to call StateHasChanged directly from a non-component thread

Avoid this:
- Don't assume dotnet is in PATH on a fresh/unfamiliar machine
- Don't make tool classes static when using the generic `WithTools<T>()` registration
- Don't keep vulnerable SK versions — NU1904/NU1903 warnings are critical

Watch out for:
- MCP SDK logging writes to stdout by default — redirect to stderr before starting the host or the protocol breaks
- TFM mismatch between SDK (build) and runtime (test execution) — test host will fail to launch with cryptic error
