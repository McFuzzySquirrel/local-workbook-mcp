# Problem / Intent

- MCP is now universally supported by Claude Desktop, GitHub Copilot, Cursor — but the server is hand-rolled and won't connect to them
- Ollama has overtaken LM Studio as the dominant local LLM runtime
- Write-back operations are now expected as a baseline
- Users want the app to work as BOTH a standalone Blazor web app AND a pluggable MCP server for any client

**Session Goal:** Define and begin implementing the true MVP across 5 phases:
1. Migrate to official MCP SDK (`ModelContextProtocol` NuGet)
2. Add write operations (cell/range write-back with file backup)
3. Ollama as default provider with auto-detection
4. MCP client integration configs (Claude Desktop, GitHub Copilot, Cursor)
5. Streaming responses in Blazor UI

# Decisions Made
- Decision: Migrate to official `ModelContextProtocol` NuGet SDK
  - Reason: Hand-rolled JSON-RPC won't connect to Claude Desktop, GitHub Copilot, Cursor, or VS Code Agent Mode
  - Impact: Unlocks all major MCP clients; replaced ~400 LOC with ~150 LOC
- Decision: Include write-back operations in MVP
  - Reason: Users expect to make changes; read-only is a significant UX limitation
  - Impact: New ExcelWriteService + 2-3 new MCP tools; file backup for safety
- Decision: Ollama as default LLM provider (LM Studio retained as fallback)
  - Reason: Dominant local runtime in 2026; Docker-friendly; free
  - Impact: appsettings.json defaults change; auto-detect logic updated
- Decision: Each MCP tool accepts `workbook_path` parameter (dual-mode design)
  - Reason: External clients (Claude Desktop, Copilot) can't pass workbook path at startup; standalone mode uses --workbook/env var
  - Impact: Tools construct ExcelWorkbookService per-call; stateless and flexible
- Decision: Blazor UI stays as standalone experience
  - Reason: MCP client configs are additive; not replacing the web app
  - Impact: Phase 4 (client configs) is purely additive, no Blazor changes
- Decision: `ExcelTools` class is `sealed` (not `static`)
  - Reason: `WithTools<T>()` SDK generic constraint disallows static types
  - Impact: Methods are logically stateless (new ExcelWorkbookService per call); minimal overhead
- Decision: MCP server logging redirected to stderr
  - Reason: MCP stdio protocol uses stdout exclusively; logging must not contaminate the protocol stream
  - Impact: `LogToStandardErrorThreshold = LogLevel.Trace` + `MinimumLevel = Warning` in host config

# Key Learnings
- `.NET 9 SDK` is not in Ubuntu 24.04 (noble) apt repos — only .NET 8 and .NET 10. .NET 10 SDK builds net9.0 TFM targets fine.
- `ModelContextProtocol` v1.1.0 SDK uses `[McpServerToolType]` on the class and `[McpServerTool(Name = "...")]` on methods. Tool class must be non-static (generic constraint in `WithTools<T>()`).
- Dual-mode workbook resolution (env var fallback + per-call param) lets the same server binary work with both `--workbook` CLI flag and external MCP clients that pass the path per tool call.
- `WithStdioServerTransport()` handles all transport framing — no need for custom Content-Length header parsing.

# If Repeating This Work
Do this:
- Check dotnet SDK availability before starting (Ubuntu noble has 8 and 10, not 9)
- Use `dotnet package search` to confirm latest NuGet versions before hardcoding
- Make tool classes `sealed` (not `static`) when using SDK `WithTools<T>()`

Avoid this:
- Don't assume dotnet is in PATH on a fresh/unfamiliar machine
- Don't make tool classes static when using the generic `WithTools<T>()` registration

Watch out for:
- MCP SDK logging writes to stdout by default — redirect to stderr before starting the host or the protocol breaks
